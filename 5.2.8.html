GIGW 3.0 Statement 5.2.8: Meaningful Sequence (WCAG 1.3.2)

Analysis and Requirements

**GIGW Statement**When the sequence in which content is presented affects its meaning, a correct reading sequence can be programmatically determined.
**WCAG Reference****Success Criterion 1.3.2 Meaningful Sequence**
**WCAG Level****Level A**
**Clause in Short Form**Correct Reading Order for Meaningful Sequence
**Goal**To ensure that the **meaning** conveyed by the order of information in the visual design is preserved for users relying on non-visual presentation.
**Requirement**When content is *sequential* (steps, dialogue, data paths), the programmatic order (in the HTML source code/DOM) must match the logical reading order.

Success Examples ✅

Success Example 1: Procedural Steps (Logical Order in Source)

Scenario: Instructions for a multi-step registration process.

  1. Fill out the personal details form. (First step in source code)
  2. Verify your email address via the link sent. (Second step in source code)
  3. Submit payment to complete registration. (Third step in source code)

**Compliance Justification:** The use of the **Ordered List (`

    `)** element and the correct **source code order** ensures that assistive technology reads the steps in the necessary logical sequence (1, 2, 3), which affects the meaning of the procedure.

Success Example 2: Conversation/Dialogue

Scenario: A short FAQ section where a question is immediately followed by its answer.

Question: How do I track my application status?

Answer: Use your tracking ID on the 'Track Status' page.

Question: What documents are required?

Answer: Aadhar card and PAN card are required.

**Compliance Justification:** The **natural order** of the content in the HTML source places the answer directly after its corresponding question. This ensures that the screen reader user hears the context *before* the information, preserving the meaning.

Failure Examples ❌

Failure Example 1: Visual Reordering via CSS

Scenario: Three sequential steps (A, B, C) are visually displayed as C, A, B using CSS `flex-order`, but the source code is A, B, C.

Visually, the steps appear as 3, 1, 2, but the source order is 1, 2, 3. (Screen reader will read 1, 2, 3).

1. Submit final application.
2. Review confirmation email.
3. Print temporary ID card.

**Compliance Justification:** The programmatic order (1, 2, 3) does not match the logical reading order (which must be 3, 1, 2 to complete the task). Relying on CSS (like `flex-order`) to change the visual sequence without changing the source order is a **Level A failure**.

Failure Example 2: Using Unordered List for Steps

Scenario: Using an unordered list (`

    `) for instructions that must be performed sequentially.

    • Place your thumb on the scanner. (Must be step 2)
    • Enter your PIN number. (Must be step 1)

    **Compliance Justification:** Although the screen reader reads the items in the source order, the use of `

      ` means it announces "List with two items, bullet" rather than "Ordered list, item 1 of 2." The **unordered semantic** incorrectly implies that the steps can be performed in any order, which affects the meaning of the procedure.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 1.3.2 Meaningful Sequence A Ensure the **order of content in the HTML source code (DOM)** exactly matches the necessary **logical sequence** for any procedural steps, conversations, or multi-part data. **Avoid using CSS or layout properties to change the visual reading order.**
