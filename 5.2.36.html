GIGW 3.0 Statement 5.2.36: Label in Name (WCAG 2.5.3)

Analysis and Requirements

**WCAG Reference****Success Criterion 2.5.3 Label in Name**
**WCAG Level****A**
**Clause in Short Form**Accessible Name Must Contain Visible Label
**Goal**To ensure that speech input users can activate any interactive component simply by speaking the text they see on the screen.
**Requirement**The text of the visible label (e.g., "Submit") must be **contained within** the programmatically determined name (e.g., "Submit Application").

Example for Success Criteria ‚úÖ

Success Example 1: Matching Names (Default Behavior)

Scenario: A simple button with clear text.

Visible Label: "Download Form"

Accessible Name: "Download Form"

**Compliance Justification:** When using standard HTML elements with visible text, the accessible name automatically matches the visible label. A user can say, **"Click Download Form"**, and the action will succeed. This is compliant.

Success Example 2: Accessible Name Contains Visible Label (Icon Button)

Scenario: A button showing only a save icon, but which provides a descriptive tooltip and accessible name.

Visible Label: "üíæ"

Accessible Name: "Save Document"

**Compliance Justification:** Although the visible label is an icon (which is essentially null text), the accessible name, **"Save Document"**, does not contain any different visible text, and it clearly communicates the control's purpose. This is compliant because the visible text (`üíæ`) is not a conflicting word. *(Note: If the icon included visible text like "Sve", the accessible name would need to contain "Sve".)*

Success Example 3: Longer Accessible Name

Scenario: A button whose visible label is enhanced by the accessible name.

Visible Label: "Submit"

Accessible Name: "Confirm and Submit Application"

**Compliance Justification:** The visible label, **"Submit"**, is **contained within** the accessible name, **"Confirm and Submit Application"**. A user can say either **"Click Submit"** or **"Click Confirm and Submit Application"** to activate the control. This is compliant.

Example for Failure Criteria ‚ùå

Failure Example 1: Conflicting Accessible Name

Scenario: A button is visually labeled "Pay Now" but programmatically labeled as "Finalize."

Visible Label: "Pay Now"

Accessible Name: "Finalize Transaction"

**Compliance Justification:** The visible text, **"Pay Now"**, is **not contained within** the accessible name, **"Finalize Transaction"**. A speech input user who says **"Click Pay Now"** will fail to activate the control, as the software is listening for the word "Finalize." This fails SC 2.5.3.

Failure Example 2: Visible Text Hidden via CSS

Scenario: A developer hides the visual text of a button to show only an icon, but the text is still the accessible name.

*(Visually appears as a button with no text, but the text "Save" is in the HTML.)*

**Compliance Justification:** If the button's purpose is not clear from the visual presentation, a user might rely on the non-visible text or assume the wrong label. While the visible text (or lack thereof) might technically be contained, misleading the user by hiding text and then forcing them to guess the hidden label fails the intent of SC 2.5.3 and other related criteria like 3.3.2 (Labels or Instructions).

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 2.5.3 Label in Name A Ensure the **words/characters displayed visibly** on any interactive component (link, button, form control) are **included** in the element's **Accessible Name** (programmatic label, often set by `aria-label`). If the visible label is "Search," the accessible name must be "Search" or "Search the Website," but **not** "Find."
