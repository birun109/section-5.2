GIGW 3.0 Statement 5.2.29: Focus Order (WCAG 2.4.3)

Analysis and Requirements

**WCAG Reference****Success Criterion 2.4.3 Focus Order**
**WCAG Level****Level A**
**Clause in Short Form**Logical Tab Order
**Goal**To ensure keyboard users can navigate interactive elements in an order that makes sense visually and semantically.
**Requirement**The order in which focus moves must follow the expected reading sequence (left-to-right, top-to-bottom) of the content.
**Developer Action**Design layouts where the **visual order** matches the **source code order**. Avoid overriding the browser's default tab sequence unless absolutely necessary for complex widgets.

Example for Success Criteria ✅

Success Example 1: Logical Form Flow (Source Code Matches Visual Order)

Scenario: A simple contact form designed in a logical, single-column flow.

*(To test, click on the **Name** field, then press the **Tab** key repeatedly. The focus should move sequentially: Name $\rightarrow$ Email $\rightarrow$ Phone $\rightarrow$ Submit.)*

**Compliance Justification:** The input fields appear in the source code in the correct logical sequence (Name, Email, Phone, Submit). By using the default tab order based on source code, the navigation sequence **preserves the meaning** of the form flow, meeting SC 2.4.3.

Example for Failure Criteria ❌

Failure Example 1: Disrupted Order using tabindex

Scenario: A form where a developer tried to manually set the focus order using high `tabindex` values, causing confusion.

*(To test, click on the **Name** field, then press the **Tab** key repeatedly. The focus jumps illogically.)*

Tab Order: 3
Tab Order: 1
Tab Order: 2

**Compliance Justification:** The visual and semantic order (Name, Email, Phone) is ignored. The keyboard user tabs: **Email (1) $\rightarrow$ Phone (2) $\rightarrow$ Name (3) $\rightarrow$ Submit (4)**. This sequence makes no sense for filling out a form, **disrupting operability and meaning**, thus failing SC 2.4.3.

Failure Example 2: Visual and Source Code Misalignment (Complex Layout)

Scenario: A multi-column layout where elements are styled to appear side-by-side but are not structured correctly in the source code.

**(Source Order 1, Visual Left)**

**(Source Order 3, Visual Right)**

**(Source Order 2, Visual Left)**

**(Source Order 4, Visual Right)**

*(If the code places Button A, B, C, D in the DOM source order, but styles it into columns, a keyboard user would tab A $\rightarrow$ B $\rightarrow$ C $\rightarrow$ D, instead of the expected reading order A $\rightarrow$ C $\rightarrow$ B $\rightarrow$ D, causing confusion.)*

**Compliance Justification:** When the visual flow (top-to-bottom, left-to-right) is A, C, B, D, but the source code order forces A, B, C, D, the **keyboard navigation order contradicts the semantic flow**. Screen readers and keyboard users will follow the illogical source order, leading to a failure of SC 2.4.3.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 2.4.3 Focus Order A Ensure that the **order of focusable elements** on a page (links, forms, buttons) is the same as the **logical and visual reading order** of the content. This is achieved primarily by structuring the HTML source code (DOM) correctly.
