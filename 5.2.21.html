GIGW 3.0 Statement 5.2.21: Keyboard (WCAG 2.1.1)

Analysis and Requirements

**GIGW Statement**All functionality of the content is operable through a keyboard interface without requiring specific timings...
**WCAG Reference****Success Criterion 2.1.1 Keyboard**
**WCAG Level****Level A**
**Clause in Short Form**All Functions Must Be Keyboard Operable
**Goal**To ensure all interactive elements can be operated by users who rely on keyboard navigation (including screen reader users, speech input users, and users with motor disabilities).
**Requirement**Every function must have a **keyboard equivalent**. The keyboard interaction must be simple (no complex timing) and must not rely solely on mouse movements.

Success Example 2: Keyboard Accessible Drag-and-Drop Alternative

Scenario: A feature that allows users to reorder items using drag-and-drop.

The page provides a "Move Up" and "Move Down" button next to each item that needs reordering.

**Compliance Justification:** While drag-and-drop inherently requires path-dependent input (mouse movement), the criterion is met because a **keyboard-accessible alternative** (**endpoint-only** functionality) is provided through the explicit "Move Up/Down" buttons. The user can achieve the same **functionality** without a mouse.

Failure Example 2: Reliance on Mouse Hover for Visibility/Action

Scenario: A critical "Next Step" button only becomes visible when the mouse cursor hovers over the parent container.

Mouse over this box to reveal the button.

**Compliance Justification:** Since keyboard focus is not equivalent to mouse hover (unless explicitly coded using `:focus` or `:focus-within`), a keyboard user may never be able to access the hidden "Next Step" button, resulting in a **loss of functionality** and failure of the criterion.
