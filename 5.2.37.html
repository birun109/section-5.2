GIGW 3.0 Statement 5.2.37: Motion Actuation (WCAG 2.5.4)

Analysis and Requirements

**WCAG Reference****Success Criterion 2.5.4 Motion Actuation**
**WCAG Level****AA**
**Clause in Short Form**Provide UI for Motion & Disable Motion
**Goal**To provide flexibility for users who cannot perform or accidentally perform device movements (e.g., due to physical disability or using a mounted device).
**Requirement**Any action triggered by device motion (shaking, tilting) must have an **equivalent UI control** AND a way for the user to **disable the motion control**.

Example for Success Criteria ✅

Success Example 1: Shake-to-Undo with Equivalent UI and Disable Option

Scenario: A mobile app allows a user to shake the phone to activate the "Undo" function.

Requirement 1: Equivalent UI Control

Requirement 2: Disable Motion Setting

Disable "Shake-to-Undo" Feature

**Compliance Justification:** The functionality ("Undo Last Action") is accessible through both the **device motion** (shaking) and an **equivalent UI control** (the "Undo Last Action" button). Additionally, a user is given the option to **disable the motion response** via a toggle switch. This satisfies both components of SC 2.5.4.

Success Example 2: Tilt-to-Scroll Functionality

Scenario: An image viewer allows tilting the device to scroll horizontally through photos.

Requirement 1: Equivalent UI Control

Requirement 2: Disable Motion Setting

The application settings contain a toggle labelled **"Disable Tilt-to-Scroll"**.

**Compliance Justification:** The scrolling function, actuated by motion (tilting), is also available via **tap controls** ("Previous/Next Photo" buttons). The presence of the disable setting ensures compliance with SC 2.5.4.

Example for Failure Criteria ❌

Failure Example 1: Motion Required, No UI Alternative

Scenario: A security authentication screen requires the user to flip the phone 180 degrees to confirm their identity.

Security Confirmation

Action Required: Please flip your device to confirm.

No button or link is provided to confirm the action manually.

**Compliance Justification:** Flipping the device is a device motion that triggers a function (confirmation). Since no alternative UI component (like a "Confirm" button) is provided, users who have their phone mounted or cannot perform the movement are excluded. This fails SC 2.5.4.

Failure Example 2: UI Alternative Provided, but Motion Cannot be Disabled

Scenario: A mapping app allows users to rotate the map by shaking the device, and a button is provided for rotation, but there is no setting to disable the shaking input.

Equivalent UI Control: **(Satisfied)**

Disable Setting: **(Missing)**

**Compliance Justification:** While the **Equivalent UI control** is present, the functionality fails because a user with a tremor who accidentally shakes their device will constantly and unwillingly trigger the map rotation. The lack of a mechanism to **disable the motion response** fails the second part of SC 2.5.4.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 2.5.4 Motion Actuation AA If your application uses **device motion** (e.g., shake, tilt) to trigger an action, you must provide **two things**: 1) An **equivalent user interface component** (a button or link) to perform the action, AND 2) A **mechanism (setting/toggle)** that allows the user to **disable** the motion input response.
