GIGW 3.0 Statement 5.2.47: Error Prevention (WCAG 3.3.4)

Analysis and Requirements

**WCAG Reference****Success Criterion 3.3.4 Error Prevention (Legal, Financial, Data)**
**WCAG Level****AA**
**Clause in Short Form**Safeguards for High-Risk Actions
**High-Risk Actions**Legal commitments, financial transactions, modifying/deleting user data, or submitting user test responses.
**Requirement**For high-risk actions, **at least one** of the following must be provided:
  • Submissions are **reversible** (e.g., an undo or cancellation option).
  • Data is **checked** for input errors, and correction opportunity is provided.
  • A **review/confirmation** step is available before finalization.

Example for Success Criteria ✅

Success Example 1: Financial Transaction with Review (Method 3)

Scenario: User is completing an online payment for service renewal.

The payment process follows three steps:

  1. Enter Details.
  2. **Review and Confirm Order** (User is shown all details, fees, and final amount on a summary page with an "Edit" option).
  3. Final Payment Submission.

**Compliance Justification:** The transaction is a high-risk financial commitment. The implementation of a mandatory **Review and Confirmation** step (Method 3) ensures the user can check and correct the information before the final action. This is compliant with SC 3.3.4.

Success Example 2: Deleting User Data with Reversibility (Method 1)

Scenario: User deletes a saved document or file from a cloud storage service.

When the user clicks "Delete," the file is moved to a **"Trash" or "Recycle Bin"** folder for 30 days and the user sees a notification stating, "File moved to Trash. You have 30 days to restore it."

**Compliance Justification:** The action involves deleting user-controllable data (high-risk). By placing the deleted data in a recoverable "Trash" folder, the action is **reversible** (Method 1). This is compliant with SC 3.3.4.

Success Example 3: Test Submission with Input Checking (Method 2)

Scenario: A user submits a government entrance exam online.

Before the final "Submit Exam" button can be clicked, the system runs an automated check, providing a summary:

**Check Result:** You have 3 unanswered questions. Please review them before submitting. (Links provided to jump to unanswered questions.)

**Compliance Justification:** The test submission is a high-risk action (submitting user test responses). The system performs **Input Checking** (for completeness) and provides the user with an opportunity to **correct** the missing data (Method 2). This is compliant.

Example for Failure Criteria ❌

Failure Example 1: Irreversible Data Deletion

Scenario: A user clicks a link to delete their entire account without any confirmation or way to undo.

When the user clicks the "Delete Account" button, a single, non-keyboard-focusable alert box appears asking, "Are you sure?" Clicking "OK" immediately executes the deletion without a review step or possibility of restoration.

**Compliance Justification:** Deleting an account is irreversible modification/deletion of user data. The single alert box is generally considered insufficient as it is too easily dismissed, and no formal review, input checking, or reversibility mechanism is provided. This fails SC 3.3.4.

Failure Example 2: Financial Transaction Without Review or Reversibility

Scenario: A government portal form for paying property tax automatically charges the entered amount upon clicking the final "Pay" button without displaying a summary page first.

The user enters their details and the amount on a single page. Clicking the **"Pay Now"** button immediately charges the user's saved card without showing a summary of the details or amount. The payment is instantly settled and non-refundable.

**Compliance Justification:** This is a high-risk financial transaction. Since there is no **Review/Confirmation** step (Method 3) and no **Reversibility** (Method 1, non-refundable), the system relies solely on input checking which may not catch all user mistakes (e.g., entering the wrong amount). This fails SC 3.3.4.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 3.3.4 Error Prevention (Legal, Financial, Data) AA **Crucial Rule for High-Risk Forms:** When an action involves a legal commitment, money, permanent data change, or test submission, provide **at least one** of these three safeguards: **Reversibility**, **Comprehensive Input Checking**, or a **Review/Confirmation Step**.
