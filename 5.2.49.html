GIGW 3.0 Statement 5.2.49: Name, Role, Value (WCAG 4.1.2)

Analysis and Requirements

**WCAG Reference****Success Criterion 4.1.2 Name, Role, Value**
**WCAG Level****A**
**Clause in Short Form**Accessible Name, Role, and State
**Goal**To ensure assistive technologies can accurately announce what a control is (Role), what it's called (Name), and its current condition (State/Value).
**Requirement**Interactive components must correctly expose:
  • **Name:** The text label or accessible name.
  • **Role:** The type of component (button, checkbox, link, etc.).
  • **State/Value:** Current status (checked, expanded, value of a text field).

Scenario: A standard checkbox where the user can set the state (checked/unchecked).

Success Example 2: Custom Dropdown Menu using ARIA

Scenario: A custom component built using `<div>` elements to act as an expandable navigation menu.

<div role="button" aria-expanded="false" aria-controls="sub-menu"> Products </div> <ul id="sub-menu">...</ul>

**Compliance Justification:** Since a `<div>` is not inherently a control, ARIA is used:

  • **Role:** `role="button"` (Tells ATs it's an interactive button).
  • **Name:** The visible text "Products".
  • **State:** `aria-expanded="false"` (Tells ATs the menu is currently closed).
When the user activates it, JavaScript must update `aria-expanded` to `"true"`, notifying the ATs of the state change. This is compliant.

Example for Failure Criteria ❌

Failure Example 1: Missing Role and State (Custom Checkbox)

Scenario: A visual checkbox created solely with a `<span>` element and CSS.

<span class="custom-checkbox"></span> <span>I accept the terms</span>

**Compliance Justification:** The `<span>` element has no inherent interactive role or state. A screen reader will only announce the visible text "I accept the terms" followed by "text." It will not recognize this as an interactive checkbox, nor will it announce its current checked state. This fails SC 4.1.2.

Failure Example 2: Custom Field Value Not Programmatically Set

Scenario: A custom input field that visualizes a number value using JavaScript but doesn't expose the value.

<div role="textbox" aria-label="Amount"> <span id="value-display">100</span> </div>

**Compliance Justification:** While the **Role** (`textbox`) and **Name** (`aria-label`) are correct, the current **Value** (`100`) is only in a generic `<span>`. For a text box role, the value must be exposed programmatically, usually through the `aria-valuenow` or by making the text content of the element available as the value. Since the value is not programmatically set or announced as the component's value, this fails SC 4.1.2.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 4.1.2 Name, Role, Value A **Rule:** For all interactive elements (native or custom), ensure they expose their **Name** (what it's called), **Role** (what it is), and **State/Value** (what it's doing/its current value) to the accessibility tree. Use **native HTML elements** whenever possible, and use **ARIA attributes** (`role`, `aria-label`, `aria-checked`, etc.) carefully and consistently for custom components.
