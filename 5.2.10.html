GIGW 3.0 Statement 5.2.10: Orientation (WCAG 1.3.4)

Analysis and Requirements

**GIGW Statement**Content does not restrict its view and operation to a single display orientation, such as portrait or landscape, unless a specific display orientation is essential.
**WCAG Reference****Success Criterion 1.3.4 Orientation**
**WCAG Level****Level AA**
**Clause in Short Form**Do Not Restrict Display Orientation
**Goal**To allow users with dexterity impairments or low vision to view content in the orientation that works best for them.
**Requirement**Content must display and operate correctly in **both portrait and landscape** modes, unless a specific orientation is **essential**.

Success Example ✅

Success Example 1: Adaptive Page Layout

Scenario: A dashboard page with a main content area and a secondary navigation panel.

Behavior: In **Landscape** (wider screens), the layout displays two columns. In **Portrait** (narrower screens, simulated below 600px), the layout stacks the secondary navigation below the main content.

Main Content Area

Application data and primary actions are displayed here. Text remains readable and functional in both orientations.

Secondary Navigation

Quick links and status summary. This moves below the main content when the orientation shifts to portrait/narrow.

**Compliance Justification:** The page uses **CSS Media Queries** to adapt the layout (from columns to stacked rows) based on the viewport dimensions. The content is fully visible and usable in both orientations, meeting the requirement.

Failure Example ❌

Failure Example 1: Restricting to Portrait Mode

Scenario: A mobile form designed only for vertical scrolling, with orientation artificially locked via a meta tag.

Application Locked to Portrait View!

If the user rotates their device to landscape, they receive an error message or the view cuts off completely. The underlying HTML/CSS prevents proper scaling.

Simulated Failure Code:
<meta name="screen-orientation" content="portrait">

**Compliance Justification:** The content forces the user into a specific orientation, which prevents users with fixed mounts (e.g., wheelchair mounts) or users wishing to zoom in on text from using their preferred mode. This is a failure because the **single orientation is not essential** for a standard form.

Essential Exception Example (Does NOT fail 1.3.4)

Scenario: An online geographical information system (GIS) map that requires landscape orientation to display the necessary full range of controls and coordinates for functional accuracy.

**Justification:** If the function of the content (e.g., highly technical data visualization or complex interactive inputs) is **fundamentally broken** or yields inaccurate results in one orientation, restriction is permissible. For most standard government content, however, restriction is **never** essential.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 1.3.4 Orientation AA Develop content using **responsive design** techniques to ensure it adapts and remains **fully usable and visible** in both **portrait and landscape** orientations. **Do not use meta tags or CSS to lock the orientation** unless the single mode is absolutely essential for the content's core function.
