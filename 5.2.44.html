GIGW 3.0 Statement 5.2.44: Error Identification (WCAG 3.3.1)

Analysis and Requirements

**WCAG Reference****Success Criterion 3.3.1 Error Identification**
**WCAG Level****A**
**Clause in Short Form**Error Identified and Described in Text
**Goal**To ensure all users, especially screen reader users, know where an error occurred and how to fix it immediately after form submission.
**Requirement**Upon automatic error detection, the field in error must be visually and programmatically identified, and a clear, descriptive text message explaining the error must be provided.

Example for Success Criteria ✅

Success Example 1: Required Field Missing

Scenario: User submits a form without filling out the email field.

Error: This field is required and cannot be blank.

**Compliance Justification:** The error field is **identified** visually (red border) and programmatically (`aria-invalid="true"`). The error is **described in text** using the associated `` element. A screen reader will announce that the field is invalid and read the error message, ensuring compliance with SC 3.3.1.

Success Example 2: Incorrect Data Format

Scenario: User enters an invalid date format.

Error: Date format is invalid. Please use DD/MM/YYYY (e.g., 01/12/2025).

**Compliance Justification:** The input field is **identified** as an error and the error is **described** clearly in text, specifying the expected format. This helps users with cognitive or learning disabilities understand how to fix the error. Compliant.

Example for Failure Criteria ❌

Failure Example 1: Error Indicated Only by Colour

Scenario: An empty required field is submitted, and the only indication of the error is a red border around the field.

**Compliance Justification:** The field is **identified** visually (red border), but the error is **not described in text**. Users who are color-blind or using screen readers will not perceive the error or know what to fix. This fails SC 3.3.1 because the description must be in text.

Failure Example 2: Error Described in Separate Location

Scenario: The error message is placed in a single, unlinked summary at the top of a long page, forcing the user to find the error manually.

Form Submission Failed:

  • Field 3 is blank.

**Compliance Justification:** While the error is technically described in text, the field itself (`id="field3"`) is not programmatically associated with the error message, and a screen reader might not link the two. More critically, the message "Field 3 is blank" is not explicitly descriptive enough, and requires the user to remember where "Field 3" is on a long form. SC 3.3.1 requires the error be identified **and** described *in text* directly related to the item in error.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 3.3.1 Error Identification A When a form error is detected: 1) **Visually and Programmatically Identify** the field in error (e.g., red border, `aria-invalid="true"`). 2) Provide a **clear, human-readable text description** of the error next to the field (e.g., "Password must contain 8 characters").
