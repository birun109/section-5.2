GIGW 3.0 Statement 5.2.24: Timing Adjustable (WCAG 2.2.1)

Analysis and Requirements

**WCAG Reference****Success Criterion 2.2.1 Timing Adjustable**
**WCAG Level****Level A**
**Clause in Short Form**Time Limits Must Be User Controllable
**Goal**To ensure users have sufficient time to interact with or read content before a session times out or a task expires.
**Requirement**One of three primary methods must be offered: **Turn off**, **Adjust** ($\ge 10\times$ default time), or **Extend** (warning $\ge 20$ seconds before expiry, extendable $\ge 10$ times).

Example for Success Criteria ✅

Success Example 1: Extend Mechanism (Most Common)

Scenario: A user is filling out a long application form with a 15-minute session timeout.

Your session will expire in 15 minutes.

At the 14:30 mark (30 seconds remaining), a modal warning pops up:

**WARNING!** Your session expires in 30 seconds. Press the **Spacebar** or **Click Here** to extend the session by 5 minutes. You can do this 9 more times.

**Compliance Justification:** This uses the **Extend** method. The user is warned at $\ge 20$ seconds before expiry and given a **simple action** (press Spacebar/Click) to extend the time. The user is allowed to extend the time at least ten times (or until the total session time is $\ge 10$ times the default). Since the user has control before the timeout, this meets the criterion.

Success Example 2: Adjust Mechanism

Scenario: A user is about to take a practice quiz where the default time limit is 5 minutes.

Before the quiz begins, the user sees a setting on the preferences page:

**Time Limit Preference:**

**Compliance Justification:** This uses the **Adjust** method. The user is allowed to adjust the time limit before encountering it to a wide range that is at least **ten times** the length of the default setting ($10 \times 5 \text{ minutes} = 50 \text{ minutes}$).

Example for Failure Criteria ❌

Failure Example 1: Abrupt Session Timeout

Scenario: A user is filling out a lengthy government application form with a 30-minute session timeout.

Session active. No time remaining display.

At the 30-minute mark, the server simply returns an "Session Expired" error, deleting all unsaved progress. **No warning was given, and no control was offered.**

**Compliance Justification:** The system fails because it does not offer any of the required controls: **Turn off**, **Adjust**, or **Extend**. The abrupt timeout without warning or opportunity to save progress creates a major accessibility barrier for users who need extra time to complete complex tasks.

Failure Example 2: Extend Limit Too Restrictive

Scenario: A user is on a transactional page with a 10-minute timeout. A warning is given, but the user can only extend the time once by 1 minute.

**Compliance Justification:** While an **Extend** mechanism is offered, it is too restrictive. The user must be allowed to extend the time limit **at least ten times** (for a total of $\ge 100$ minutes in this case, or until $10\times$ the default time is reached). Allowing only one small extension fails the "Extend" criteria and thus fails the SC 2.2.1.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 2.2.1 Timing Adjustable A For any content-set time limit (e.g., session timeouts, reading time), developers must provide a way for the user to control the duration. The preferred method for session timeouts is the **Extend** option: warn the user $\ge 20$ seconds before expiry and offer an easy way to refresh the session $\ge 10$ times. If using **Adjust**, the max time must be $\ge 10 \times$ the default time.
