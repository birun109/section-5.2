GIGW 3.0 Statement 5.2.48: Parsing (WCAG 4.1.1)

Analysis and Requirements

**WCAG Reference****Success Criterion 4.1.1 Parsing**
**WCAG Level****A**
**Clause in Short Form**Valid and Well-Formed Markup
**Goal**To guarantee that browsers and assistive technologies can process and interpret the page content consistently and without crashing.
**Requirements**
  • **Proper Tags:** Elements requiring both start and end tags must have them (e.g., `<div>` and `</div>`).
  • **Correct Nesting:** Elements must be nested in the correct order (e.g., a paragraph inside a list item).
  • **Unique IDs:** Every `id` attribute value must be unique on the page.
  • **No Duplicated Attributes:** Attributes cannot be repeated on the same element.

Example for Success Criteria ✅

Success Example 1: Correct Tag Nesting and ID Usage

Scenario: Proper structure of a header element with a navigation list.

<header id="main-header"> <h1>Ministry Portal</h1> <nav role="navigation"> <ul> <li><a href="/">Home</a></li> <li><a href="/contact">Contact</a></li> </ul> </nav> </header>

**Compliance Justification:** All elements have complete tags and are correctly nested (e.g., `<a>` inside `<li>`, `<li>` inside `<ul>`). The `id` attribute (`main-header`) is unique. This well-formed markup is compliant with SC 4.1.1.

Success Example 2: Self-Closing Tags

Scenario: Using elements that do not require closing tags, as allowed by HTML5 specifications.

<label for="name">Name:</label> <input type="text" id="name"> <br> <img src="logo.png" alt="Logo">

**Compliance Justification:** Elements like `<input>`, `<br>`, and `<img>` do not require explicit closing tags (`</input>`, etc.) under the current HTML5 specification. This adherence to the specification is compliant with SC 4.1.1.

Example for Failure Criteria ❌

Failure Example 1: Incorrect Nesting

Scenario: Overlapping or incorrectly nested HTML tags.

<p>This is <strong>bold and <em>italic.</strong></em></p>

**Compliance Justification:** The `<strong>` tag is closed before the `<em>` tag, even though `<em>` was opened after `<strong>`. The correct nesting should be `<strong><em>...</em></strong>`. This overlapping structure confuses the parser, leading to unpredictable rendering or accessibility issues. This fails SC 4.1.1.

Failure Example 2: Duplicate IDs and Attributes

Scenario: Using the same ID multiple times and duplicating an attribute on one element.

<div id="menu">...</div> <div id="menu">...</div> <!-- Duplicate ID --> <p id="footer" style="color:red" style="font-size:large">...</p> <!-- Duplicate Attribute: style -->

**Compliance Justification:** The `id` attribute is meant to be a unique identifier for scripting and linking. Using `id="menu"` twice is a severe parsing error. Similarly, having the `style` attribute twice on the `<p>` element is invalid markup. Both issues violate the "well-formed" requirement and fail SC 4.1.1.

Summary of Conformity

WCAG Reference WCAG Level Key Action for Compliance
SC 4.1.1 Parsing A **Crucial Rule:** Always ensure your HTML/markup is **valid and well-formed**. Use a formal W3C HTML Validator to check for errors like missing end tags, incorrect nesting, duplicate attributes, and, most importantly, non-unique `id` attributes.
